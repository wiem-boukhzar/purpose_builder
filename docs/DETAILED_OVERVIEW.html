Purpose Creator – Comprehensive Technical Guide

Deep-dive into the Streamlit, LangGraph, and Disease Ontology powered agent for pharma workflows.

1. System Landscape

This overview combines the architectural tiers and highlights the responsibilities of each layer.

User / Stakeholder

  • Pharma researchers entering disease concepts through chat.
  • Regulatory/product teams consuming the exported JSON payload.

Streamlit UI (frontend/app.py / frontend/app.py / …)

  • Chat display, input form, auto-scroll, and “option N” shortcuts.
  • Structured summary panel with validation messages and downloads.

Purpose Application Layer

  • LangGraph workflow orchestrating generate → parse → validate nodes.
  • Logging utilities (`purpose_app/logging_utils.py`) emitting JSON telemetry.

Ontology & Validation Services

  • Ontology loader + label index (synonyms, alt labels, exact matches).
  • RapidFuzz-based disease linker, SHACL validator, and subclass suggestions.
  • Pre-computed child map for low-latency validation in no-SHACL mode.

Data, Config & Infra

  • data/ontology/doid.owl snapshot and SHACL shapes.
  • Python dependencies (`requirements.txt`, set up in a fresh virtual environment).
  • Environment variables for Apollo/OpenAI access and logging path overrides.

2. Entry Points & Runtime Options

Four Streamlit entrypoints cover hosted vs. local deployment and SHACL vs. pre-computed validation.

File Description LLM Endpoint Validation Mode
frontend/app.py Hosted Streamlit UI Capgemini OpenAI-compatible SHACL
frontend/app.py Local UI Model endpoint SHACL
Hosted UI without SHACL Capgemini endpoint Pre-computed child map
Local UI without SHACL Apollo API Pre-computed child map

Session State Highlights

  • messages – stores canonical user/assistant content plus optional display string (for “option 4” style replies).
  • ontology_* keys – track linked concepts, outstanding generic parents, and unknown terms with suggestions.
  • validation_cached / validation_locked / validated_diseases_key – prevent repeated SHACL evaluations for unchanged disease lists.
  • option_shortcuts – keeps the currently displayed numeric suggestions so the next reply can reference them.

3. Purpose Application Core

Core workflow logic lives in purpose_app/common.py. Streamlit files orchestrate UI layout and delegate all processing.

3.1 LangGraph Workflow

The conversation loop is implemented as a simple LangGraph pipeline whose nodes mutate session_state.

generate → call_*_model
  • Compose instructions + full conversation context for the LLM.
  • Emit llm_request/llm_response telemetry.
  • Store the raw model reply in raw_response.
parse → coerce_to_schema
  • Extract the JSON payload from the raw completion.
  • Normalise diseases, yes/no answers, medication strings.
  • Log parse_error if deserialisation fails.
validate → run_disease_validation
  • RapidFuzz linking against the ontology index.
  • SHACL specificity check (or child-map shortcut) with caching.
  • Populate linked/unknown/generic lists and emit validation_result* logs.
end → merge
  • LangGraph END node returns a diff merged into session_state.
  • Streamlit renders the next assistant prompt and structured summary.
  • Option shortcuts enable quick replies like “option 4”.

3.2 Essential Helpers

run_disease_validation

Links diseases, runs SHACL (or child-map) checks, populates session state, and emits validation_result logs.

build_assistant_reply

Guides the conversation: prompts for subtypes, lists remaining fields, and registers numeric shortcuts for quick selections.

build_purpose_payload

Generates the final JSON export capturing DO IDs, labels, and user-supplied metadata (medication, collaborator, product development, TA).

resolve_option_shortcut

Maps “option 3” style replies to the underlying suggestion before invoking the LangGraph pipeline.

3.3 Telemetry

  • purpose_app/logging_utils.py sets up a rotating JSON log (override location with PURPOSE_LOG_PATH).
  • Emitted events:
    • chat_message – role, canonical content, display text (if any).
    • llm_request / llm_response – provider, model, message count, status, error payload.
    • parse_error – truncated snippet when JSON extraction fails.
  • Logs can be forwarded to Splunk/ELK/Grafana for observability and alerting.

4. Ontology & Validation Layer

The ontology layer handles RDF parsing, label indexing, fuzzy matching, and specificity validation.

4.1 Data Flow

Loading, indexing, and validating diseases flows through the ontology layer as a staged pipeline:

1
Ontology bootstrap

load_graph() parses doid.owl, caches the RDF graph, and records version info.

2
Index construction

build_label_index() normalises English labels, alt labels, and exact synonyms; build_child_map() records subclass existence for fast lookups.

3
Fuzzy linking

link_many() maps user disease strings to LinkedDisease (IRI/label/score) or UnknownDisease (original text + suggestions).

4
Specificity validation

Default path: validate_diseases() runs SHACL (RDFS inference + SPARQL). Optional path: HAS_CHILDREN lookup offers near-instant results.

5
Subtype suggestions

suggest_children() queries direct subclasses of umbrella concepts so the chat can present numbered refinements.

4.2 SHACL Rule Summary

  • ex:ResearchIntentShape – requires at least one ex:diseaseConcept.
  • ex:DiseaseSpecificityShape – SPARQL constraint: “?child rdfs:subClassOf $this” flags umbrella classes.

4.3 SHACL vs. Pre-computed Child Map

Aspect SHACL Pre-computed child map
Runtime cost Higher (RDFS inference + SPARQL) Minimal (dictionary lookup)
Extensibility Declarative — add new shapes without code Requires bespoke Python/SPARQL changes
Auditability Standard SHACL report graph Custom logs only
Best suited for Regulated, compliance-focused deployments Rapid prototyping or latency-sensitive demos

5. Testing Strategy

Run tests via python -m pytest inside an activated virtual environment. They operate on the real Disease Ontology snapshot.

Test Coverage
tests/test_linker.py Abbreviations (NSCLC) and ambiguous inputs (diabetes → diabetes mellitus).
tests/test_suggestions.py Subclass suggestions (lung cancer variants).
tests/test_validator.py SHACL specificity (diabetes mellitus fails, subtype passes).
tests/test_smoke.py Smoke scenarios (multiple diseases, misspellings, unknown terms).

6. Operational Guidance

6.1 Deployment Tips

  • Create a fresh virtual environment (e.g., python -m venv .venv) and install dependencies with the bundled requirements.txt.
  • Provide APOLLO_CLIENT_ID / APOLLO_CLIENT_SECRET (and optional APOLLO_TOKEN_URL, APOLLO_BASE_URL, APOLLO_MODEL) for Apollo access.
  • Ensure logs/ is writable (or set PURPOSE_LOG_PATH) to persist JSON telemetry.
  • For enterprise rollout, front the UX with corporate SSO or migrate the workflow into a backend service that Streamlit calls.

6.2 Suggested Next Steps

  • Ship logs to a central observability stack for monitoring latency, unknown-term frequency, and validation errors.
  • Automate nightly smoke tests to detect ontology or prompt regressions.
  • Extend the SHACL shapes as new business rules emerge (e.g., therapeutic area consistency checks).
  • Consider trimming the ontology with ROBOT if memory footprint or startup latency becomes problematic.
