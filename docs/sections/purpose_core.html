Purpose Application Core

A closer look at purpose_app/common.py and the helper modules that orchestrate extraction, validation, and responses.

Design Goals

  • Deterministic pipeline – every user message flows through generate → parse → validate before the UI updates.
  • State-driven UI – Streamlit components read the same session_state mutated by the LangGraph nodes.
  • Observability – each stage emits JSON events via logging_utils.log_event.

Key Modules

LangGraph nodes

build_graph(call_model) assembles the generate, parse, and validate nodes. Each node returns a partial ResearchState update that is merged into session_state.

Normalisation helpers

normalise_diseases, normalise_text, and normalise_yes_no ensure the structured payload is clean and deduplicated before storage.

Reply logic

build_assistant_reply decides whether to prompt for subtypes, clarify unknowns, or move on to medication/collaborator/product development questions.

Export helpers

build_purpose_payload generates the JSON artifact (diseases with DOIDs, TA, medication, collaborator, product development, ontology metadata).

Example: Single Turn

# user submits "type 2 diabetes mellitus"
add_message("user", "type 2 diabetes mellitus")
result = graph.invoke({"user_messages": [... canonical history ...]})
update_structured_data(result["structured"])
validation = result["validation_result"]  # linked vs unknown
assistant_reply = build_assistant_reply(st.session_state.structured_data)
add_message("assistant", assistant_reply)

This sequence executes on every turn, keeping business logic isolated from UI code so the workflow can be reused in services or unit tests.
