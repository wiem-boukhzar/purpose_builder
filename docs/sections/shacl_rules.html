SHACL Rule Summary

These shapes power the default specificity validation. They can be extended with additional business constraints if needed.

Shapes File

@prefix sh:   <http://www.w3.org/ns/shacl#> .
@prefix rdfs: <http://www.w3.org/2000/01/rdf-schema#> .
@prefix ex:   <http://example.org/schema#> .

ex:ResearchIntentShape a sh:NodeShape ;
  sh:targetClass ex:ResearchIntent ;
  sh:property [
    sh:path ex:diseaseConcept ;
    sh:minCount 1 ;
    sh:nodeKind sh:IRI
  ] .

ex:DiseaseSpecificityShape a sh:NodeShape ;
  sh:targetObjectsOf ex:diseaseConcept ;
  sh:severity sh:Violation ;
  sh:message "Disease is generic; please pick a more specific subtype." ;
  sh:sparql [
    a sh:SPARQLConstraint ;
    sh:select "SELECT $this WHERE { ?child rdfs:subClassOf $this . } LIMIT 1"
  ] .

Extending the Shapes

  • Add additional sh:property constraints to require therapeutic area consistency or custom datatype checks.
  • Use sh:severity to distinguish warnings vs. hard violations.
  • Bundle the shapes with your build artifact so compliance teams can version-control validation rules.
